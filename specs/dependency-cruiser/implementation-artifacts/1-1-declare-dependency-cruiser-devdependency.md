# Story 1.1: Declare dependency-cruiser as a devDependency

Status: draft

## Story

As a maintainer,
I want dependency-cruiser declared as a pinned devDependency installable through Bun,
so that the graph-hygiene check is available through the existing docker-compose bun tooling
without any new runtime install..

## Acceptance Criteria

1. `dependency-cruiser` is present in `package.json` `devDependencies` with the version range
   `^17.3.7`.
2. `dependency-cruiser` is declared only as a devDependency — never under `dependencies`,
   `peerDependencies`, or `optionalDependencies`.
3. `bun.lock` records the resolved `dependency-cruiser` version after Bun resolves the lockfile.
4. `bun x depcruise --version` runs inside the docker-compose `bun` service without any separate
   install step.
5. No `scripts` entry is introduced in `package.json` for dependency-cruiser; the repository
   stays script-free by convention.
6. The tool is invoked only through Bun (`bun x depcruise`), never through an npm/package.json
   script or a globally installed binary.

## Tasks / Subtasks

- [ ] Task 1: Add `dependency-cruiser` to `package.json` devDependencies (AC: 1, 2, 5)
  - [ ] 1.1 Add `"dependency-cruiser": "^17.3.7"` to the `devDependencies` object in
        `package.json`, keeping keys alphabetically ordered with the surrounding entries
  - [ ] 1.2 Confirm the entry does NOT appear under `dependencies`, `peerDependencies`, or
        `optionalDependencies`
  - [ ] 1.3 Confirm `package.json` still has no `scripts` block (do not add one for
        dependency-cruiser)

- [ ] Task 2: Resolve and commit the lockfile (AC: 3, 4)
  - [ ] 2.1 Bring the toolchain up with `make start` so the docker-compose `bun` service is
        running
  - [ ] 2.2 Install via the `bun` service so Bun resolves and writes `bun.lock`
        (e.g. `$(RUN_BUN_SH) 'bun install'`), keeping the install inside the container
  - [ ] 2.3 Confirm `bun.lock` now records a resolved `dependency-cruiser` entry satisfying
        `^17.3.7`
  - [ ] 2.4 Commit both `package.json` and `bun.lock` together so the lockfile stays in sync

- [ ] Task 3: Confirm Bun-only invocation works in the service (AC: 4, 6)
  - [ ] 3.1 Run `$(BUN_X) depcruise --version` inside the `bun` service and confirm it prints a
        `17.x` version without triggering a fresh install
  - [ ] 3.2 Confirm the binary is reachable as `bun x depcruise` (the `BUN_X = $(BUN) x` path),
        not through any host-global install or `package.json` script

- [ ] Task 4: Verification (AC: 1-6)
  - [ ] 4.1 Inspect `package.json` and confirm `dependency-cruiser: ^17.3.7` is present only
        under `devDependencies`
  - [ ] 4.2 Confirm `package.json` contains no `scripts` key
  - [ ] 4.3 Confirm `bun.lock` contains the resolved `dependency-cruiser` version
  - [ ] 4.4 Run `make start` then `$(BUN_X) depcruise --version` and confirm a clean
        non-error exit, then `make down`

## Dev Notes

### Architecture Decisions (from specs/dependency-cruiser/planning-artifacts/architecture-dependency-cruiser-2026-06-23.md)

This story implements **only Decision 1: Tool & Version**. The `.dependency-cruiser.js` policy
file (Decision 2/3), the `lint-dep-cruiser` Makefile target (Decision 4), and the
`dependency-cruiser.yml` CI workflow (Decision 5) are out of scope here and are delivered by
later Epic 1 / Epic 2 stories. This story makes the tool _available_, not yet _enforced_.

- **Consume as a devDependency, invoke through `bun x`:** dependency-cruiser is declared in
  `package.json` `devDependencies` and run as `bun x depcruise` inside the existing
  docker-compose `bun` service — no separate binary download, no runtime install, no system path.
- **Pinned range:** `^17.3.7`, matching the CRM pin and the repository's caret dependency-range
  policy. Bun (`bun@1.3.5`) manages resolution; the resolved version is captured in `bun.lock`.
- **No `package.json` scripts:** the repository deliberately keeps `package.json` script-free;
  the Makefile is the command source of truth. This story adds a dependency line only — it
  introduces no `scripts` entry. The Makefile bun invocation chain is
  `RUN_BUN = docker compose run --rm bun`, `BUN = $(RUN_BUN) bun`, `BUN_X = $(BUN) x`, so the
  later gate will run as `$(BUN_X) depcruise ...`.
- **Affected files (this story):** `package.json` (new devDependency) and `bun.lock` (resolved
  entry) only.

### Project Structure Notes

- **Files to modify:** `package.json` (add the `dependency-cruiser` devDependency) and
  `bun.lock` (regenerated by Bun to record the resolved version).
- **No new files** are created in this story; `.dependency-cruiser.js`,
  the `Makefile` `lint-dep-cruiser` target, and `.github/workflows/dependency-cruiser.yml`
  are introduced by later stories.
- `package.json` currently has a `devDependencies` block and intentionally no `scripts` block —
  preserve both characteristics.
- All install/resolution work must happen inside the docker-compose `bun` service via the
  Makefile bun helpers (`make start` / `$(RUN_BUN_SH) 'bun install'` / `make down`); do not run
  a host-level install.

### Testing Approach

This story is dependency-declaration only, so there is no Jest unit test to add. Verification is
shell/inspection based (Task 4) inside the docker-compose `bun` service:

- Inspect `package.json`: `dependency-cruiser: ^17.3.7` present under `devDependencies` only, and
  no `scripts` key.
- Inspect `bun.lock`: a resolved `dependency-cruiser` entry satisfying `^17.3.7`.
- Smoke-run availability: `make start` then `$(BUN_X) depcruise --version` prints a `17.x`
  version and exits cleanly without a fresh install, then `make down`.

The `lint-dep-cruiser` Makefile target and the dependency-cruiser CI workflow do not exist yet,
so no `make lint-dep-cruiser` assertion is part of this story; that arrives once Story 1.2+
commit the policy file and Epic 2 adds the workflow.
